# Claude 함대 컨트롤 — 설계 문서

- 작성일: 2026-07-21
- 상태: Phase 1 스펙 (전체 로드맵은 맥락으로 포함)

## 1. 문제 / 목표

여러 프로젝트에서 Claude Code 세션을 동시에 굴리는데, 지금은 프로젝트마다 터미널을 열고 직접 붙어서 확인·응답해야 한다. 세션이 많아지니 관리가 안 된다.

원하는 것:
- 세션들이 **auto 모드로 자율 진행**하고,
- **진짜 결정이 필요할 때만** 맥락을 스스로 설명해서 한곳에 모아 보여주고,
- 나는 **1·2·3 번호 / 메모 / 음성**으로 답하면 그 답이 해당 세션에 들어가 이어서 일하게.
- 세션은 **진짜 터미널로 떠 있어야** 하고(직접 타이핑·종료 가능), 패널에서 닫으면 터미널도 닫히고, 새로/`/resume` 로 띄울 수 있어야 한다.
- 프로젝트당 **최대 2 세션**.
- 결정 패널은 **폰에서, 집 밖에서도** 접속.

## 2. 확정된 결정 사항

| 항목 | 결정 |
|---|---|
| 세션 구동 방식 | **A안** — 네이티브 `claude` (tmux 안) + MCP 결정 브릿지 |
| 호스트 머신 | 이 데스크톱 맥 (상시 실행, repo 보유) |
| 폰 접속 | **집 밖에서도** → Tailscale (맥 + 폰) |
| 터미널 실체 | tmux 창의 진짜 `claude` 프로세스 |
| auto 모드 | `--permission-mode acceptEdits` + 함대 규칙 주입 |
| 프로젝트당 세션 | 최대 2 |
| 프로젝트 위치 | `~/Desktop/claude-fleet` (git) |

## 3. 전체 비전 (로드맵 맥락)

```
[프로젝트별 claude 세션들]  ──결정필요──▶  [오케스트레이터 프로세스]  ──▶  [웹 패널(폰)]
   tmux 안에서 auto모드로 일함           - MCP 결정 서버(블로킹)       1·2·3 / 메모 / 음성
        ▲                               - tmux 세션 레지스트리             │
        └──────────  답을 툴 리턴값으로 세션에 주입 → 이어서 일함  ◀────────┘
```

**한 개의 오케스트레이터 프로세스**(맥에서 상시)가 (1) 웹 패널 서빙, (2) MCP 결정 서버 호스팅, (3) tmux 세션 관리 + 레지스트리를 모두 담당한다. 폰은 Tailscale로 이 프로세스의 웹 패널에 접속한다.

**동작 흐름:** 세션이 되돌리기 힘든/외부영향/제품 갈림길을 만나면 네이티브 질문창에서 멈추지 않고 `request_decision` 툴을 호출한다. 이 툴콜은 오케스트레이터에서 **응답이 올 때까지 블로킹**되고, 그 사이 카드가 패널에 뜬다. 내가 답하면 답이 **툴 리턴값으로 세션에 주입**되어 세션이 이어서 일한다.

## 4. Phase 1 범위 (이번 구현 대상)

핵심 루프를 얇게 수직으로 증명한다: **세션이 물음 → 폰에 카드 → 탭 → 세션이 이어감.**

포함:
- 오케스트레이터 프로세스 골격 (웹 + MCP + 결정 큐)
- `request_decision` MCP 툴 (블로킹 + 리턴)
- 최소 웹 패널: 카드(제목·지금왜·payoff·트레이드오프·맥락) + 1·2·3 버튼 + 메모 입력
- Tailscale 경유 폰 접속 + 패널 접근 토큰
- 세션 1개를 tmux로 수동 실행하는 부트스트랩 절차 (auto 모드 + 함대 규칙 + MCP 연결)
- 결정 히스토리 로그(JSONL)

제외(→ Phase 2/3):
- 오케스트레이터의 세션 런처/`/resume`/닫기 자동화, 최대2 강제, 생사 동기화
- 카드 풀스타일링, 음성 입력, 파일 첨부, 자동 저장, 히스토리 UI

## 5. 아키텍처 / 컴포넌트

각 컴포넌트는 단일 책임을 갖고 독립적으로 테스트 가능하게 나눈다.

### 5.1 오케스트레이터 프로세스 (Node + TypeScript, 단일 프로세스)
- 기술: Node.js + TypeScript. 최소 의존성 — `@modelcontextprotocol/sdk`(MCP 서버), 내장 `http`(또는 경량 서버), tmux는 `child_process`.
- 하나의 프로세스가 두 개의 HTTP 표면을 연다:
  - `/mcp` — MCP streamable HTTP 엔드포인트 (세션들이 붙는 곳)
  - `/` , `/api/*` — 웹 패널 + 패널 API
- 상태: 인메모리 `pendingDecisions` 맵 + 파일 기반 결정 로그.

### 5.2 결정 브릿지 (MCP 서버)
- 툴 하나: `request_decision`. 입력 스키마(§6). 핸들러 동작:
  1. 결정 id 생성, `pendingDecisions[id] = { resolve, payload, sessionToken, createdAt }`.
  2. 패널에 알림(SSE 또는 폴링으로 노출).
  3. 응답이 올 때까지 **await** (Promise). 타임아웃은 매우 길게(기본 없음) — 사람이 답할 때까지 대기.
  4. 패널이 답을 주면 Promise resolve → 결과를 툴 리턴값으로 반환 + 히스토리에 append.
- 세션 식별: 세션 실행 시 발급한 **세션 토큰**을 MCP 연결(URL 쿼리/헤더)에 실어, 어느 프로젝트/세션의 결정인지 태깅. (Phase 1은 단일 세션이라 토큰 1개; 구조는 다중 대비.)

> Phase 1 구현은 이 MCP 서버를 오케스트레이터에 붙는 별도 **stdio 브릿지**로 두고, 세션→오케스트레이터는 로컬 HTTP 롱폴링으로 잇는다(HTTP 트랜스포트의 세션관리 복잡도 회피).

### 5.3 웹 패널 (Phase 1 최소)
- 단일 페이지 + 소량 JS. 폰 화면 우선(큰 버튼).
- 표시: pending 결정 카드 — 제목, 지금 왜?, THE PAYOFF, 트레이드오프, 알아두면 좋은 맥락, 옵션 버튼(1·2·3, 각 action_preview), 메모 입력.
- 동작: 버튼/메모 → `POST /api/decisions/:id/answer` → 해당 툴콜 resolve.
- 갱신: 새 결정 도착은 SSE(`/api/stream`) 또는 짧은 폴링.
- 접근 제어: 패널 토큰(쿼리/쿠키) 1개. 없는 요청은 거절.

### 5.4 세션 부트스트랩 (Phase 1은 문서화된 명령 + 얇은 헬퍼)
tmux 창에서 다음을 만족하는 `claude` 실행:
- 함대 MCP 서버 연결 (세션 토큰 포함한 mcp config)
- auto 모드: `--permission-mode acceptEdits`
- 함대 규칙 주입: `--append-system-prompt "<규칙>"` (프로젝트 파일 안 건드림)
- 규칙 요지: *"자율로 계속 일하되, 되돌리기 힘든/외부영향/제품 갈림길에선 멈추지 말고 `request_decision`으로 맥락(지금왜·payoff·트레이드오프·옵션) 채워 올려라. 자잘한 확인은 스스로 판단해 진행."*

### 5.5 안전망 (Notification 훅)
- 세션이 라우팅되지 않은 네이티브 권한창 등에서 멈추면, Notification 훅이 오케스트레이터에 "세션 X 멈춤"을 POST → 패널에 표시. (놓침 방지. Phase 1은 최소 표시만.)

## 6. 결정 프로토콜 — `request_decision` 스키마

```jsonc
// 입력 (세션 → 오케스트레이터)
{
  "title": "완주 API 인증 없이 열려 있음 — 이메일 선물 켜기 전 서명검증 넣을지",  // 상황 한 줄
  "why_now": "이메일 켜는 순간 '가짜 완주→진짜 선물 발송' 통로가 됨...",            // 지금 왜?
  "payoff": "이메일 켜기 전에 문 잠그기 🔒 vs 나중에 악용 발견하고 급하게 막기 🔥", // 이걸 정하면 뭐가 달라지나
  "tradeoff": "서명검증 추가는 반나절 규모 엔지니어링 / 미루면 노출 지속",         // 트레이드오프
  "context": "누가 API 직접 호출해 임의 사용자를 완주 표시 가능한 pre-existing 취약점", // 알아두면 좋은 맥락(옵션)
  "options": [
    { "n": 1, "label": "지금 서명검증 넣고 이메일 켜기", "action_preview": "🔒 가짜 완주로 선물 빼가는 길 원천 차단" },
    { "n": 2, "label": "이메일 먼저 켜고 서명검증은 다음", "action_preview": "..." },
    { "n": 3, "label": "완주 API 자체를 인증 뒤로 옮기기", "action_preview": "..." }
  ],
  "allow_freetext": true
}

// 리턴 (오케스트레이터 → 세션)
{
  "choice": 1,                         // 고른 번호(없을 수 있음)
  "memo": "1번. 대신 배포 전 한 번 더 확인해줘",  // 자유 메모(없을 수 있음)
  "files": []                          // Phase 3
}
```

- 카드의 각 섹션은 이 필드에 1:1 매핑되어, mockup UI가 그대로 렌더된다.
- 세션은 규칙에 따라 이 필드들을 **스스로 채워** 풍부한 카드를 만든다.

## 7. 데이터 흐름 (Phase 1)

```
claude 세션 ──request_decision(툴콜)──▶ MCP핸들러
   (블로킹 대기)                          │ pendingDecisions[id]=…  + SSE로 패널 알림
        ▲                                 ▼
   툴 리턴값 ◀──resolve(answer)── POST /api/decisions/:id/answer ◀── 폰 패널(버튼/메모)
```

## 8. 원격 접속 (Tailscale)
- 맥과 폰에 Tailscale 설치, 같은 tailnet.
- 폰은 `http://<맥의-tailscale-이름 또는 IP>:<포트>/?token=…` 로 패널 접속.
- 포트폴딩·공개 노출 없음. 패널 토큰으로 2차 방어.

## 9. 상태 / 저장
- `pendingDecisions`: 인메모리 (id → {resolve, payload, sessionToken, createdAt}).
- 결정 히스토리: `~/Desktop/claude-fleet/data/decisions.jsonl` 에 append (질문 + 답 + 타임스탬프).
- 세션 레지스트리(프로젝트→세션 목록, 최대2): Phase 2. Phase 1은 토큰 1개.

## 10. 에러 처리
- 답 대기 중 세션이 죽으면: 해당 pending은 만료 표시(패널에서 "세션 종료됨"), resolve 시도 시 무시.
- 패널이 이미 답한 결정에 또 답하면: id가 pending에 없으므로 409/무시.
- 오케스트레이터 재시작: 인메모리 pending은 소실 → 블로킹 중이던 툴콜은 에러 리턴(세션은 그걸 받고 재시도/보고). 히스토리 파일은 유지.
- 잘못된/없는 패널 토큰: 401.

## 11. 테스트 전략 (TDD)
- **단위:** `request_decision` 입력 스키마 검증; 결정 큐 enqueue→answer→resolve 흐름; 패널 토큰 가드.
- **통합:** 오케스트레이터 기동 → 가짜 MCP 클라이언트가 결정 등록(블로킹) → 패널 API로 답 → 툴콜이 그 답으로 resolve되는지 검증; 히스토리 append 검증.
- **수동 E2E:** 실제 `claude` 세션을 tmux로 띄우고, `request_decision`을 트리거하는 프롬프트 실행 → 폰(Tailscale)에서 카드 확인·응답 → 세션이 이어지는지 확인.

## 12. Phase 1 완료 기준 (수용 기준)
1. 오케스트레이터가 `localhost:<포트>` 에 패널, `/mcp` 에 MCP를 연다.
2. tmux의 실제 `claude` 세션이 함대 MCP + auto 모드로 붙고, 지시 시 `request_decision`을 호출해 블로킹된다.
3. 폰(Tailscale)에서 패널을 열면 카드가 제목·지금왜·payoff·트레이드오프·맥락 + 1·2·3 + 메모로 보인다.
4. 옵션 탭 또는 메모 전송 시 세션의 툴콜이 그 답으로 리턴되고 세션이 이어서 일한다.
5. 결정이 `decisions.jsonl` 에 기록된다.

## 13. Phase 2 / 3 로드맵
- **Phase 2 — 함대 관리:** 오케스트레이터 런처(띄우기/`claude --resume <id>`/닫기), 프로젝트당 최대2 강제, 세션 레지스트리, 패널의 세션 목록·런처, tmux 생사 동기화(패널에서 닫으면 kill, 직접 닫으면 감지).
- **Phase 3 — 카드/입력 완성:** mockup 풀스타일, 음성 입력(브라우저 Web Speech API), 파일 첨부, 자동 저장, 결정 히스토리 UI.

## 14. 기본값 / 열린 항목
- 포트: 기본값 1개 정해 설정 가능(예: 4179).
- 프론트: Phase 1 은 순수 HTML/JS 단일 페이지, Phase 3 에서 정식 스타일.
- 다중 세션 식별 토큰 방식(쿼리 vs 헤더): 구현 시 MCP SDK 지원에 맞춰 확정.
